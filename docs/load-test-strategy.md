# MCJ 부하테스트 전략 및 실행 정책

## 1. 개요

이 문서는 MCJ 서비스의 핵심 사용자 흐름을 기준으로 부하테스트 범위, 우선순위, 실행 환경, 관측 지표, 결과 기록 방식을 정의합니다.

목표는 다음과 같습니다.

- 배포 이후 실제 트래픽 증가 전 병목 가능 구간을 사전에 식별합니다.
- 조회성/쓰기성/배치성 흐름별로 다른 부하 특성을 구분해 검증 기준을 정리합니다.
- 시나리오별 테스트 순서와 결과 기록 기준을 일관되게 맞춥니다.
- 추후 결제 모니터링 전용 대시보드 및 도메인 메트릭과 연계할 수 있는 확인 기준을 마련합니다.

## 2. 범위

이번 부하테스트는 특정 API 단건 성능 측정이 아니라, 실제 사용자 행동과 운영 흐름을 기준으로 한 시나리오 검증을 목표로 합니다.

### 2.1 조회성 흐름

- 공동구매 목록 조회
- 공동구매 상세 조회
- 마이페이지 참여/주문/환불 조회
- 관리자/사장님 대시보드 및 운영 조회

### 2.2 쓰기성 흐름

- 결제 주문 생성
- 결제 승인
- 결제 취소
- 찜 등록/해제

### 2.3 상태 전이 및 외부 연동 흐름

- 결제 웹훅 처리
- PortOne 연동 포함 결제 승인/취소 흐름
- 상태 전이 후속 처리

### 2.4 배치성 흐름

- 환불 처리 후속 작업
- 상태 전이/후처리 스케줄러

## 3. 1차 우선 검증 범위

1차 검증은 실제 사용자 영향도가 높고, 병목 발생 시 장애 체감이 큰 흐름을 우선 대상으로 합니다.

### 3.1 최우선

- 공동구매 목록 조회
- 공동구매 상세 조회
- 결제 주문 생성
- 결제 승인

### 3.2 후속 우선순위

- 결제 취소
- 결제 웹훅 처리
- 마이페이지 조회
- 관리자/사장님 조회
- 환불/배치성 처리

## 4. 환경 정책

### 4.1 기본 원칙

- 운영 환경(`prod`) 직접 부하테스트는 기본 수행 대상에서 제외합니다.
- 부하테스트는 staging 성격 환경이 있으면 해당 환경을 우선 사용합니다.
- staging 성격 환경이 없으면 제한된 `dev` 환경에서 수행하되, 테스트 목적 트래픽이 일반 개발 트래픽과 구분되도록 운영합니다.

### 4.2 dev 환경 사용 시 제약

- 테스트 시간대를 사전 고정합니다.
- 테스트 데이터를 별도로 준비합니다.
- 일반 개발 작업과 동시에 대규모 부하를 유입하지 않습니다.
- 테스트 트래픽 식별 기준을 별도로 둡니다.
- 테스트용 모니터링 알림 채널 또는 라우팅 분리를 검토합니다.
- `develop` 브랜치 기준 변경사항이 dev 환경에 배포된 이후 실행합니다.

### 4.3 외부 연동 정책

- 외부 결제 연동이 포함된 시나리오는 실제 외부 API 직접 호출 여부를 단계적으로 나눠 검토합니다.
- 초기 시나리오 검증 단계에서는 대체 응답 또는 제한된 샌드박스 환경 활용 가능성을 우선 검토합니다.
- 실제 외부 연동 포함 테스트는 비용, 제한 정책, 알림 영향, 멱등성 위험을 사전에 확인한 뒤 수행합니다.

## 5. 실행 원칙

### 5.1 테스트 유형

- 단일 API 집중 부하
- 핵심 사용자 흐름 기반 시나리오 부하
- 점진적 증가(ramp-up) 테스트
- 순간 동시 요청(spike) 테스트
- 장시간 지속(soak) 테스트

### 5.2 시나리오 작성 원칙

- 정상 흐름과 오류 흐름을 분리합니다.
- 인증이 필요한 API는 실제 사용자 흐름과 유사한 인증 단계를 반영합니다.
- 상태 전이가 있는 흐름은 사전 데이터 상태를 명시합니다.
- 후속 처리나 웹훅이 연결되는 시나리오는 선행 조건과 완료 조건을 함께 기록합니다.

### 5.3 테스트 도구 원칙

- 스크립트 기반 반복 실행과 결과 수집이 가능한 도구를 사용합니다.
- 1차 후보는 `k6`로 검토합니다.
- 도구 선정 시 CI 연계 가능성, 시나리오 유지보수성, 결과 비교 편의성을 함께 고려합니다.

## 6. 관측 기준

부하테스트 수행 시 다음 지표를 공통 확인 대상으로 둡니다.

### 6.1 애플리케이션 공통 지표

- 평균 응답시간
- p95 / p99 응답시간
- 처리량(RPS/TPS)
- 에러율
- HTTP 4xx / 5xx 비율
- timeout 발생 여부

### 6.2 인프라 및 런타임 지표

- CPU 사용률
- 메모리 사용률
- JVM Heap 사용량
- GC pause
- DB/Redis 등 연관 자원 병목 징후

### 6.3 도메인 관측 지표

- 결제 주문 생성 성공/실패
- 결제 승인 성공/실패
- 결제 취소 성공/실패
- 결제 웹훅 처리 성공/실패
- 환불/후처리 성공/실패

### 6.4 모니터링 연계 기준

- 기존 Prometheus/Grafana 패널에서 확인 가능한 지표를 우선 활용합니다.
- 결제 모니터링 관련 작업이 머지되면 결제 전용 대시보드 기준으로 관측 항목을 재정렬합니다.
- 테스트 수행 전후로 새로 필요한 패널 또는 알람 분리 요구사항을 기록합니다.
- dev 부하테스트 수행 전 dev 메트릭 수집 범위와 dev 대시보드 노출 여부를 먼저 확인합니다.

## 7. 결과 기록 기준

모든 테스트는 아래 항목을 동일한 형식으로 남깁니다.

- 테스트 일시
- 대상 환경
- 대상 시나리오
- 사전 데이터 조건
- 동시 요청 수 / 증가 방식 / 지속 시간
- 평균 응답시간 / p95 / p99
- 실패율 / 5xx / timeout
- 병목 의심 지점
- 로그/대시보드 관찰 결과
- 후속 개선 필요사항

## 8. 구현 순서 기준

부하테스트 구현과 실행은 다음 순서로 진행합니다.

1. 조회성 흐름 baseline 시나리오 작성
2. 결제 주문 생성/승인 시나리오 작성
3. 결제 취소/웹훅 시나리오 작성
4. 마이페이지/운영 조회 시나리오 작성
5. 배치성 흐름 검증 범위 확정
6. 결과 기록 템플릿 및 운영 확인 가이드 정리
7. 결제 모니터링 작업 머지 후 대시보드/지표 연계 보완

## 9. 제외 범위

현재 단계에서 아래 항목은 즉시 구현 대상으로 보지 않습니다.

- 운영 환경 대상 직접 대규모 부하 실행
- 실서비스 사용자 데이터에 영향을 줄 수 있는 무제한 결제 연동 테스트
- 부하테스트 결과를 기준으로 한 자동 스케일링 정책 수립
- 모든 배치성 작업의 동시 검증

## 10. 후속 작업

이 문서를 기준으로 다음 작업을 순차 진행합니다.

- 부하테스트 디렉터리 구조 및 공통 실행 설정 추가
- 핵심 시나리오별 스크립트 작성
- 테스트 결과 기록 템플릿 추가
- 환경별 모니터링/알림 분리 필요사항 반영
